# ChangingRoom — Architecture

## What the app does

ChangingRoom is a virtual try-on tool. The user uploads two images — a photo of themselves and a clothing item — and the app returns a Gemini-generated image of them wearing it.

---

## Full request flow

```
Browser
  │
  │  User selects two images → clicks "Try It On"
  │
  ▼
UploadForm.tsx  (Client Component — runs in the browser)
  │
  │  fetch("POST /api/tryon", FormData { person, clothing })
  │
  ▼
Next.js server  (Node.js process running locally)
  │
  │  app/api/tryon/route.ts receives the request
  │  Reads GOOGLE_AI_API_KEY from .env.local
  │  Converts File objects → base64 strings
  │
  ▼
Google Gemini API  (gemini-2.5-flash-image model)
  │
  │  Returns a candidate with an inlineData image part (base64 PNG)
  │
  ▼
route.ts  (server, continued)
  │
  │  Wraps the base64 bytes in a data: URL
  │  Returns JSON { imageUrl: "data:image/png;base64,..." }
  │
  ▼
UploadForm.tsx  (browser, continued)
  │
  │  Passes imageUrl to <ResultDisplay />
  │
  ▼
ResultDisplay.tsx
  │
  │  Renders <img src={imageUrl} /> → user sees the generated outfit
  ▼
  Done
```

---

## Server vs. client — why the split matters

Next.js App Router has two kinds of components:

**Server Components** (default — no `"use client"` at the top)
- Rendered on the server at request time.
- Cannot use React hooks (`useState`, `useEffect`) or browser APIs.
- Never sent as interactive JS to the browser — only the resulting HTML is sent.
- `page.tsx` and `layout.tsx` are Server Components.

**Client Components** (`"use client"` at the top of the file)
- Rendered on the server first (for the initial HTML), then *hydrated* in the browser so they become interactive.
- Can use hooks and browser APIs.
- `UploadForm.tsx` and `ResultDisplay.tsx` are Client Components because they need `useState`, `useRef`, and `fetch`.

The API route (`route.ts`) is neither — it is a pure server-side handler, like an Express endpoint. It never runs in the browser at all. This is where the Gemini API key is used, so it stays secret.

---

## How Next.js App Router routing works

File path → URL. That's the rule.

| File | URL |
|---|---|
| `app/page.tsx` | `GET /` |
| `app/api/tryon/route.ts` | `POST /api/tryon` |

There is no explicit router config. Next.js scans the `app/` directory and treats every `page.tsx` as a page and every `route.ts` as an API endpoint. Adding a new file automatically creates a new route.

## Every file and what it does

### Application source

| File | Role |
|---|---|
| `app/layout.tsx` | **Root layout.** Wraps every page in `<html lang="en"><body>`. Required by Next.js App Router — there must be exactly one root layout. Exports `metadata` (sets the browser tab title and meta description). |
| `app/page.tsx` | **Home page.** A Server Component that renders the static page shell (heading, footer) and embeds `<UploadForm />`. |
| `app/globals.css` | **Global styles.** One line: `@import "tailwindcss"` — this is all Tailwind v4 needs. Also sets the default font on `body`. |
| `app/api/tryon/route.ts` | **API route.** Receives a `POST` with `FormData`, converts images to base64, calls the Gemini API, returns `{ imageUrl }` or an error JSON. The Gemini key never leaves the server. |
| `components/UploadForm.tsx` | **Client Component.** Manages two image slots with hidden `<input type="file">` elements. On submit, sends a `fetch` POST to `/api/tryon` and passes the result to `<ResultDisplay>`. |
| `components/ResultDisplay.tsx` | **Client Component.** Receives three props — `loading`, `result`, `error` — and renders the appropriate UI: a CSS spinner, an error card, or the result image with a download link. |

### Configuration files

| File | Role |
|---|---|
| `package.json` | Lists the 4 runtime dependencies (`next`, `react`, `react-dom`, `@google/genai`) and the 5 dev-only ones (TypeScript, Tailwind, type definitions). Also defines the `dev`, `build`, and `start` scripts. |
| `package-lock.json` | Exact snapshot of every installed package and its transitive dependencies. Checked into git so that `npm install` always produces the same `node_modules` on any machine. |
| `next.config.ts` | Next.js configuration entry point. Empty by default, but Next.js won't start without it. Add custom headers, rewrites, or env vars here when needed. |
| `postcss.config.mjs` | Tells Next.js's build pipeline to run PostCSS on CSS files, with `@tailwindcss/postcss` as the plugin. Tailwind v4 compiles its utility classes through PostCSS — without this file, Tailwind classes would not generate any CSS. |
| `tsconfig.json` | TypeScript compiler settings. Key options: `"strict": true` (catches more bugs), `"jsx": "preserve"` (Next.js handles JSX transform), `"moduleResolution": "bundler"` (supports modern import syntax). |
| `next-env.d.ts` | Auto-generated by Next.js on first run. Gives TypeScript knowledge of Next-specific types (`NextRequest`, `NextResponse`, image imports, etc.). Never edit this manually. |
| `.env.local` | Stores `GOOGLE_AI_API_KEY`. Loaded by Next.js automatically — variables are available via `process.env` in server-side code. Never sent to the browser. Never committed (listed in `.gitignore`). |
| `.gitignore` | Tells git what not to track: `.env*` (secrets), `.next/` (build output), `node_modules/` (reinstallable), `tsconfig.tsbuildinfo` (TypeScript cache). |

## Why each dependency exists

| Package | Why it's needed |
|---|---|
| `next` | The framework: handles routing, server rendering, API routes, the dev server, and the production build. |
| `react` | The UI library: provides JSX, hooks (`useState`, `useRef`), and the component model. |
| `react-dom` | React's adapter for the browser DOM. Separate from `react` so React can target other platforms (React Native, etc.). |
| `@google/genai` | The current Google AI SDK. Used in `route.ts` to call the Gemini image generation API. |
| `tailwindcss` | Utility-first CSS framework. Instead of writing CSS files, you compose classes like `flex`, `rounded-2xl`, `text-zinc-500` directly in JSX. |
| `@tailwindcss/postcss` | PostCSS plugin that compiles Tailwind's utility classes into real CSS at build time. |
| `typescript` | Adds static types to JavaScript. Catches type mismatches at compile time instead of at runtime. |
| `@types/node` | TypeScript type definitions for Node.js built-ins (`process.env`, `Buffer`, etc.). |
| `@types/react` | TypeScript type definitions for React (`React.ReactNode`, `ChangeEvent`, etc.). |
| `@types/react-dom` | TypeScript type definitions for React DOM. |
